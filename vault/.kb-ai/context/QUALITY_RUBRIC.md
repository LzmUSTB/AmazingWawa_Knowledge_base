# Quality Rubric

Target quality is clear, technically correct, reusable, and at least as instructive as the source material.

## Authoring rules

- Prefer one node at a time for note generation.
- Prefer one node at a time for ExerciseSet generation.
- Preserve a consistent style based on previous user feedback.
- Self-check questions must include answers.
- Avoid learner-facing meta prose about AI, package generation, snapshots, locale compliance, or implementation.
- Preserve important source visuals, explanations, and interactions.

## Note review

Check definition, problem solved, intuition, mechanism, formal detail, examples, mistakes, related knowledge, source attribution where needed, and review answers. Verify text, tables, controls, canvases, and code blocks in the actual application theme.

## ExerciseSet review

Check target node, scope ids, unique problem ids, legal type and difficulty values, complete prompt/answer/solution fields, and appropriate academic depth.

## Package review

Before package delivery, validate the exact root files, `mimetype`, manifest schema, object-shaped `patch.yaml`, canonical operations, referenced generated files, asset paths, and required source-review files. Package generation remains explicit-only.
