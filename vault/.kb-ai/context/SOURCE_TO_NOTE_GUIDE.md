# Source-to-Note Guide

## Analyze before authoring

Extract reusable knowledge, teaching structure, source assets, important demonstrations, claims, parameters, and candidate nodes. Do not summarize paragraphs in source order.

For each candidate node, record why it should exist and reject items that are too small, duplicated, or source-specific. Propose hierarchy and semantic relations separately.

## Source inventory

For visual or interactive sources, inventory important images, videos, animations, interactive demos, downloadable projects, and source snapshots. Record source URL/location, teaching role, preservation strategy, target node, and whether the item is required.

## Note plan

Choose `markdown` for compact static concepts, `html` for source-rich or interactive tutorials, and `none` for intentional placeholders. Preserve the source's strongest teaching devices without turning the note into implementation commentary.

Use nearby source blocks for claims, media, or demonstrations that actually derive from source material. Do not force source blocks into original explanations.

## Approval gate

After source analysis, produce an import plan first. Include proposed and rejected nodes, relations, note formats, asset treatment, ExerciseSet plan, and quality risks. Do not generate package files until the user approves the plan.
